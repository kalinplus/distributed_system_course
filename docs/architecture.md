# 系统架构设计

## 1. 概述

### 1.1 背景

本系统面向电商场景，构建一个简化的「用户–商品–库存–订单」业务闭环，用于支撑课程中"高并发读""分布式缓存""读写分离""负载均衡"等技术点的综合实践。

### 1.2 目标

- 提供基础的用户注册登录、商品浏览、下单购买等功能。
- 在高并发访问商品列表和商品详情的场景下，通过缓存、读写分离、负载均衡等手段保障良好的响应时间和系统稳定性。
- 演示基于 Docker / docker-compose 的容器化部署，以及 Nginx 反向代理、动静分离、多实例负载均衡。
- 为后续"商品库存与秒杀"扩展预留架构空间。

## 2. 需求分析

### 2.1 功能需求

#### 2.1.1 用户模块

- **用户注册**
  - 用户可以通过账号（用户名或手机号）+ 密码注册。
  - 注册时校验账号唯一性。
- **用户登录/登出**
  - 用户通过账号密码登录，登录成功后建立会话（Session 或 Token）。
  - 用户可以主动登出，清除会话。
- **用户信息查询（简单版，可选）**
  - 查询当前登录用户的基础信息。

#### 2.1.2 商品模块

- **商品管理（简化，可通过后台页面或初始化脚本）**
  - 创建、编辑商品：名称、价格、库存数量、描述、状态（上架/下架）等。
- **商品列表**
  - 分页查询商品列表，支持按关键字或分类简单筛选（可选）。
- **商品详情**
  - 根据商品 ID 返回详细信息，是高并发读的主要入口。
  - 详情数据支持 Redis 缓存。

#### 2.1.3 库存模块

- **库存展示**
  - 在商品详情及商品列表中展示当前可售库存。
- **库存扣减**
  - 用户下单时，校验库存并扣减库存。
  - 当前阶段使用数据库本地事务保证订单与库存的一致性。
- **库存查询**
  - 根据商品 ID 查询库存信息（供内部服务调用）。

#### 2.1.4 订单模块

- **创建订单**
  - 登录用户选择商品及数量后提交订单。
  - 下单时校验库存是否充足，扣减库存并生成订单记录。
- **订单状态管理**
  - 至少包括：`待支付`、`已支付`、`已取消` 等状态。
  - 支付可以使用模拟流程：例如直接标记为已支付或通过简单回调接口。
- **订单查询**
  - 用户查看自己的订单列表。
  - 用户查看单个订单详情。

### 2.2 非功能需求

#### 2.2.1 性能与并发

- **并发场景**
  - 商品详情页和商品列表的高并发读请求。
  - 通过 JMeter 模拟数百至一两千 QPS 级别的并发访问。
- **性能目标（课程级）**
  - 引入 Redis 缓存和 MySQL 读写分离后：
    - 商品详情接口平均响应时间显著低于仅访问数据库的方案。
    - 主库压力明显降低，大部分读流量由从库和缓存承担读取。
  - 在 Nginx 负载均衡下，后端多个实例的请求量大致均衡。

#### 2.2.2 可用性与稳定性

- 高并发读压测下，系统持续可用，不出现大量错误响应或进程崩溃。
- 针对缓存穿透、缓存击穿、缓存雪崩提供防护措施（**已实现**）：
  - 缓存空值（`"null"` 字符串，TTL=5min），避免缓存穿透。
  - 逻辑过期 + SETNX 分布式互斥锁 + 异步刷新线程，避免缓存击穿。
  - TTL 随机抖动（30min ± 5min），避免缓存同时失效造成雪崩。
- 简单降级方案：缓存或从库故障时，可退化为主库直读或返回部分数据，并记录日志。

#### 2.2.3 一致性要求

- **读一致性**
  - 商品详情和商品列表采用"读性能优先 + 最终一致性"策略：
    - 商品信息更新后，缓存和从库数据允许短时间不一致。
- **事务一致性**
  - 订单创建与库存扣减在当前阶段采用单库本地事务，保证强一致。
- **缓存一致性策略**
  - 对商品信息更新时，采用"先更新数据库，再删除或更新缓存"的策略，保证最终收敛一致。
  - 文档中明确说明可能出现的短暂不一致场景及原因。

#### 2.2.4 安全与访问控制

- 登录态校验：只有登录用户才允许下单、查看订单。
- 基础参数校验与权限校验，防止未授权访问他人订单。
- 简单防刷（可选）：可在 Nginx 或应用层实现基于 IP / 用户 ID 的限流策略。

#### 2.2.5 部署与运维

- 使用 Docker / docker-compose 管理数据库、缓存、Nginx、后端服务容器。
- 使用 Nginx 实现反向代理、动静分离与负载均衡。
- 日志：
  - 应用层记录关键业务日志和异常。
  - Nginx 记录访问日志，用于统计各后端实例请求分布和响应时间。

## 3. 系统架构设计

### 3.1 整体架构描述

整体采用前后端分离 + 多服务拆分架构，核心组件如下：

- **前端**
  - 简单的 HTML/JS 页面或轻量前端框架，用于商品浏览和下单操作。
  - 静态资源由 Nginx 提供动静分离访问。
- **API 网关 / 反向代理**
  - 使用 Nginx 作为统一入口：
    - 将 `/api/**` 流量转发到后端多个应用实例（负载均衡）。
    - 将静态资源请求转发到静态目录或前端容器。
- **后端服务**
  - 采用微服务风格的逻辑拆分，按功能划分为：
    - `用户服务`：用户注册、登录、用户信息。
    - `商品服务`：商品列表、商品详情、商品管理。
    - `库存服务`：库存查询、库存扣减。
    - `订单服务`：订单创建、订单查询、订单状态管理。
  - 在实现上可以共享一个代码工程，通过模块/包划分服务边界，也可以拆成多个独立服务（视课程要求和实现成本而定）。
- **数据存储**
  - **MySQL 主从集群**：主库负责写操作（注册、下单、库存更新），从库负责读操作（商品查询、订单查询等），实现读写分离。
  - **Redis 缓存**：缓存商品详情、热门商品列表等读操作结果。采用三重防护策略（穿透/击穿/雪崩）。
  - **ElasticSearch**：用于商品搜索，存储商品索引数据。服务启动时从 MySQL 全量同步。

### 3.2 服务拆分说明

#### 3.2.1 用户服务（User Service）

- 职责：
  - 用户注册、登录、登出。
  - 用户信息查询。
- 对外暴露能力（简要说明，具体 API 后续单独文档）：
  - 用户注册接口。
  - 用户登录/登出接口。
  - 获取当前用户信息接口。

#### 3.2.2 商品服务（Product Service）

- 职责：
  - 商品信息的增删改查。
  - 对外提供商品列表和详情查询接口。
  - 维护商品与库存、订单之间的基础关联。
  - 提供 ElasticSearch 全文搜索能力。
- 对外暴露能力：
  - 获取商品列表。
  - 获取商品详情（整合三重缓存防护逻辑）。
  - 商品关键词搜索（基于 ElasticSearch）。
  - （后台）新增/编辑商品。

**缓存三重防护机制：**

| 防护类型 | 问题 | 方案 | 实现 |
|----------|------|------|------|
| 穿透防护 | 查询不存在的数据，缓存永远 miss | 缓存空值 `"null"` | `setNullValue()` — 短 TTL 5 分钟 |
| 击穿防护 | 热点 Key 过期瞬间，大量请求同时击穿 DB | 逻辑过期 + SETNX 互斥锁 | `CacheData<T>` 包装 + `acquireLock()` + 异步刷新线程 |
| 雪崩防护 | 大量缓存同一时刻过期 | TTL 随机抖动 | `calculateTtlWithJitter()` — 30min ± 5min |

缓存请求流程：
1. 查询 Redis 中的 `CacheData<Product>` 结构
2. 若缓存不存在 → 回源查库 → 写入逻辑过期缓存
3. 若缓存存在且未过期 → 直接返回（P50=5ms）
4. 若缓存存在但已过期 → SETNX 抢锁：
   - 抢到锁的线程：启动异步线程刷新缓存，当前请求返回过期数据
   - 未抢到锁的线程：短暂等待后重查缓存

**读写分离机制：**

- 双 HikariCP 数据源：HikariPool-1 (Master:mysql:3306) + HikariPool-2 (Slave:mysql-slave:3306)
- `AbstractRoutingDataSource` 根据 ThreadLocal 上下文路由到不同数据源
- `@ReadOnly` 注解标记读方法，`ReadOnlyRoutingAspect` AOP 切面自动设置路由标记
- 读操作（`selectById`）→ 路由到 Slave；写操作 → 路由到 Master

#### 3.2.3 库存服务（Inventory Service）

- 职责：
  - 维护商品库存数据。
  - 对外提供库存查询、库存预扣减/扣减等能力。
- 对外暴露能力：
  - 查询商品库存。
  - 扣减库存（为订单服务调用）。
- 实现要点：
  - 与数据库交互时使用本地事务，保证库存扣减与订单创建的一致性。
  - 后续可扩展为独立服务，支持更复杂的库存预留、异步扣减、秒杀等。

#### 3.2.4 订单服务（Order Service）

- 职责：
  - 订单创建、订单状态迁移。
  - 订单查询。
- 对外暴露能力：
  - 创建订单接口（内部会调用库存服务扣减库存）。
  - 查询用户订单列表。
  - 查询订单详情。
- 实现要点：
  - 创建订单时，与库存服务组合形成一个小的本地事务场景（初期可采用单库事务实现）。

## 4. 部署架构

### 4.1 容器化部署

使用 Docker / docker-compose 管理各组件：

- **MySQL 主从集群**：主库 + 从库容器
- **Redis**：缓存容器（三重防护）
- **ElasticSearch**：搜索容器（单节点）
- **Nginx**：反向代理 + 负载均衡容器
- **后端服务**：多个应用实例容器

### 4.2 负载均衡

- Nginx 作为统一入口，实现：
  - 反向代理：转发 API 请求到后端多个实例
  - 负载均衡：轮询或加权轮询
  - 动静分离：静态资源直接由 Nginx 提供
